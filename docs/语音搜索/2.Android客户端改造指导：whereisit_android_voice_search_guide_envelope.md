# WhereIsIt Android 客户端语音搜索改造指导

## 1. 文档目的

本文档用于指导 WhereIsIt Android 客户端增加一期语音搜索功能。目标是在不重做现有搜索页的前提下，为当前 Kotlin + Jetpack Compose + ViewModel + Repository + Retrofit 架构加入：

1. 语音搜索入口
2. 按住说话页面
3. 实时文本回显
4. 松开后触发最终识别与搜索
5. 复用现有物品搜索结果页展示结果

Android 客户端 README 已说明项目使用 Jetpack Compose、ViewModel、Repository 分层、Retrofit 网络请求封装，并在 `app/src/main/java/com/whereisit/findthings/` 下按 `data/` 与 `ui/` 分层组织代码。 citeturn781242view3turn781242view7

## 2. 改造目标

## 2.1 Envelope 适配要求（新增）

服务端语音搜索相关 **HTTP 接口返回必须符合 Envelope 规范**，因此 Android 客户端新增接口时，不应直接把最终响应体定义为裸 `VoiceFinalizeResponse`，而应定义为：

- `Envelope<VoiceFinalizeResponse>`
- 或复用项目现有通用响应包装类型

建议抽象：

```kotlin
data class Envelope<T>(
    val code: Int,
    val message: String,
    val data: T?
)
```

约束建议：

- `code == 0` 视为成功
- `data == null` 时要按失败处理，不能直接进入结果页
- 语音搜索的 `finalText/items/timing` 等字段都位于 `Envelope.data` 内
- WebSocket partial 消息不必强制套用 Envelope，仍按实时消息模型处理即可


一期 Android 端只负责：

1. 录音和交互
2. 与服务端流式 STT 通信
3. 实时展示 partial 文本
4. 松开后触发最终识别与搜索
5. 展示已有结果页

一期 Android 端**不负责**：

- 本地语音识别
- 本地最终搜索
- 复杂同音纠错
- 复杂分词或字典管理

也就是说：

- **实时文本来自服务端第一阶段**
- **最终查询结果来自服务端第二阶段 + 搜索**

## 3. 建议新增的客户端结构

在现有结构基础上，建议新增以下模块。

```text
app/src/main/java/com/whereisit/findthings/
  data/
    voice/
      VoiceSearchApi.kt
      VoiceSearchWebSocketClient.kt
      VoiceRecorder.kt
      VoiceSearchRepository.kt
      model/
        VoiceWsMessage.kt
        VoiceFinalizeRequest.kt
        VoiceFinalizeResponse.kt
        Envelope.kt
  ui/
    voice/
      VoiceSearchRoute.kt
      VoiceSearchScreen.kt
      VoiceSearchViewModel.kt
      VoiceSearchUiState.kt
```

职责如下：

### `VoiceRecorder.kt`
- 封装 `AudioRecord`
- 输出 PCM 音频帧
- 控制开始/停止录音

### `VoiceSearchWebSocketClient.kt`
- 建立 WebSocket
- 发送 `start/audio/stop`
- 接收 partial 文本

### `VoiceSearchApi.kt`
- 定义 `POST /api/voice-search/finalize`
- 返回 `Envelope<VoiceFinalizeResponse>`

### `VoiceSearchRepository.kt`
- 对上层暴露统一接口：
  - `startVoiceSearch()`
  - `sendAudioFrame()`
  - `stopAndFinalize()`

### `VoiceSearchViewModel.kt`
- 管理页面状态
- 协调录音、WebSocket、finalize 请求

### `VoiceSearchScreen.kt`
- Compose 页面
- 呈现按住说话、实时文本、状态提示

## 4. 交互设计建议

## 4.1 入口位置

建议从现有搜索入口进入，例如：

- 搜索框右侧麦克风图标
- 搜索页上的“语音搜索”按钮

## 4.2 页面文案

进入语音搜索页后，固定显示：

- 标题：`语音搜索`
- 提示：`按住并说出物品名称或品牌`

## 4.3 核心交互

### 初始状态
- 麦克风按钮可按下
- 页面显示提示文案
- 文本区域为空或显示“等待输入”

### 按住状态
- 用户按住按钮开始录音
- 按钮进入高亮/按下态
- 文案变成“正在聆听，请继续说话”
- 页面实时显示 partial 文本

### 松开状态
- 用户松开按钮
- 停止录音
- 按钮恢复正常态
- 页面显示“正在识别并搜索”
- 等待最终结果

### 完成状态
- 成功：进入或刷新现有搜索结果页
- 失败：显示错误提示并允许重试

## 5. 录音实现建议

## 5.1 基础参数

建议统一为：

- sample rate: 16000
- channels: mono
- encoding: PCM 16bit

与服务端保持一致，减少转换成本。

## 5.2 录音模式

建议使用“按住录音”的手势，而不是点击开始/点击停止。原因：

1. 更符合短指令搜索场景
2. 更容易控制音频长度
3. 用户能明确知道何时结束输入

## 5.3 `VoiceRecorder` 设计示意

```kotlin
interface VoiceRecorder {
    fun start(onAudioFrame: (ByteArray) -> Unit)
    fun stop()
}
```

建议内部逻辑：

1. 创建 `AudioRecord`
2. 在后台协程/线程持续读取 buffer
3. 把 buffer 按固定大小回调出去
4. stop 时释放资源

### 5.4 帧大小建议

可以按 20ms ~ 100ms 音频块发送。不要太小，否则 WebSocket 消息过多；不要太大，否则实时性下降。

建议初版可按约 100ms 一帧实现。

---

## 6. WebSocket 实时识别接入

服务端会暴露一个实时识别 WebSocket。客户端需要完成：

1. 建立连接
2. 发送 `start`
3. 边录音边发 `audio`
4. 接收 `partial`
5. 松开后发送 `stop`

## 6.1 `VoiceSearchWebSocketClient` 建议接口

```kotlin
interface VoiceSearchWebSocketClient {
    suspend fun connect()
    suspend fun sendStart(sampleRate: Int = 16000)
    suspend fun sendAudio(seq: Int, pcm: ByteArray)
    suspend fun sendStop()
    fun observeMessages(): Flow<VoiceWsMessage>
    suspend fun close()
}
```

## 6.2 partial 文本更新策略

服务端返回 partial 时，客户端不要做太重的业务修正，只做轻量 UI 处理：

- 去前后空白
- 防止重复刷新完全相同的文本
- 用最新 partial 覆盖上一条 partial

### 不建议客户端做的事情

- 大量正则纠错
- 本地分词
- 本地品牌推断
- 本地搜索

因为一期里，最终文本以第二阶段为准。

---

## 7. finalize 请求接入

当用户松开按钮后，客户端需要：

1. 停止录音
2. 发送 `stop`
3. 进入 loading 状态
4. 调 `POST /api/voice-search/finalize`
5. 获取最终文本和搜索结果
6. 跳转到现有结果页

## 7.1 `VoiceSearchApi` 示例职责

```kotlin
interface VoiceSearchApi {
    suspend fun finalizeVoiceSearch(
        request: VoiceFinalizeRequest
    ): Envelope<VoiceFinalizeResponse>
}
```

如果后端使用 `session_id` 模式，则请求体只需要：

- `sessionId`
- `firstStageText`（可选）

如果后端使用直接上传音频模式，则在 Repository 层改成 multipart 上传。

---

## 8. ViewModel 状态机建议

语音搜索页最适合做成明确状态机。

## 8.1 UI 状态定义建议

```kotlin
sealed interface VoiceSearchUiState {
    data object Idle : VoiceSearchUiState
    data class Listening(val partialText: String) : VoiceSearchUiState
    data class Finalizing(val lastText: String) : VoiceSearchUiState
    data class Success(val finalText: String) : VoiceSearchUiState
    data class Error(val message: String, val lastText: String?) : VoiceSearchUiState
}
```

## 8.2 状态流转

- `Idle`
  - 按下 -> `Listening`
- `Listening`
  - 收到 partial -> 更新 partialText
  - 松开 -> `Finalizing`
- `Finalizing`
  - finalize 成功 -> `Success`
  - finalize 失败 -> `Error`
- `Error`
  - 点击重试 -> `Idle`

### 8.3 为什么要显式状态机

因为语音页天然有录音中、等待中、完成、失败等状态，显式状态机会比一堆布尔值更稳定。

## 9. Compose 页面设计建议

## 9.1 页面布局建议

从上到下：

1. 顶部标题栏
2. 提示文案
3. 实时文本显示区
4. 录音按钮区
5. 状态提示或错误提示

## 9.2 文本显示区

建议区分两类文本：

- 第一阶段 partial：灰色或普通样式
- 第二阶段 final：成功后可在结果页中显示“你说的是：xxx”

一期语音页本身只需要展示 partial 和 loading 即可。

## 9.3 按住按钮

建议使用 `pointerInput` 或手势检测，支持：

- `onPress` 开始录音
- `onRelease` 停止录音
- `onCancel` 异常终止录音

### 一期交互要求

- 按下即开始
- 松开即停止
- 手势取消也要 stop，避免资源泄漏

---

## 10. 与现有搜索结果页的衔接

一期不要新做一个“语音结果页”，而是：

1. `VoiceSearchViewModel` 拿到 finalize 响应
2. 将响应中的 `items` 转换为现有结果页使用的数据结构
3. 直接导航到已有的搜索结果页
4. 让结果页像处理文字搜索一样处理数据

如果当前结果页依赖 query 参数，则可以：

- 把 `finalText` 作为显示 query
- 把 `items` 作为已加载结果传给结果页

如果当前结果页只能根据 query 再去请求一次，也可以只传 `finalText`，让现有搜索逻辑重跑一次。但从用户体验看，更推荐直接复用 finalize 返回的结果。

---

## 11. 权限与异常处理

## 11.1 录音权限

需要申请：

- `RECORD_AUDIO`

建议流程：

1. 进入语音页前检查权限
2. 没有权限则弹系统申请
3. 用户拒绝时给出清晰文案

### 提示文案建议

- `需要麦克风权限才能进行语音搜索`

## 11.2 网络异常

常见错误：

- WebSocket 建立失败
- finalize 超时
- 服务端识别失败

建议 UI 文案：

- `实时识别连接失败，请重试`
- `语音识别超时，请再说一次`
- `未识别到清晰的物品名称，请重试`

## 11.3 录音异常

- 麦克风被占用
- AudioRecord 初始化失败
- 用户快速点按导致 start/stop 竞态

要在 Repository 或 ViewModel 层兜底，确保：

- stop 可重复调用而不崩溃
- 页面退出时一定释放录音与 WebSocket

## 12. Repository 层实现建议

建议新增 `VoiceSearchRepository` 统一封装整个流程：

```kotlin
class VoiceSearchRepository(
    private val recorder: VoiceRecorder,
    private val wsClient: VoiceSearchWebSocketClient,
    private val api: VoiceSearchApi,
) {
    suspend fun startListening()
    suspend fun stopAndFinalize(): VoiceFinalizeResponse
    fun partialTextFlow(): Flow<String>
}
```

### 12.1 Repository 内部职责

- 控制录音开始/停止
- 控制 WebSocket 连接状态
- 转发 partial 文本到 ViewModel
- 调用 finalize API
- 屏蔽底层实现细节

这样 ViewModel 只负责状态管理，不直接操作录音和网络细节。

---

## 13. 一期不建议在客户端做的事

以下能力不建议放到一期 Android 端：

1. 本地 sherpa-onnx 模型推理
2. 本地 FunASR 推理
3. 本地分词与关键词提取
4. 本地品牌字典管理
5. 复杂实时纠错
6. 录音缓存长期持久化

原因：

- 复杂度高
- 容易和服务端逻辑分叉
- 影响调试效率
- 一期目标只是打通语音搜索通路

## 14. 埋点建议

建议客户端埋以下事件：

- 打开语音页
- 按下录音按钮
- 松开录音按钮
- 实时 partial 次数
- finalize 成功/失败
- 结果页展示成功
- 用户点击了哪个物品

这些事件有助于判断：

- 用户是否愿意使用语音搜索
- partial 回显是否足够快
- 第二阶段成功率如何
- 结果页是否有实际转化

## 15. 实施顺序建议

### 步骤 1：做页面壳子
- 新建 `VoiceSearchScreen`
- 完成文案、按钮、状态区

### 步骤 2：接入录音
- 完成按住录音 / 松开停止

### 步骤 3：接入 WebSocket
- 能显示 partial 文本

### 步骤 4：接入 finalize
- 能拿到最终结果

### 步骤 5：接入现有结果页
- 能展示 items 列表

### 步骤 6：补异常处理
- 权限、超时、重试、页面退出释放资源

## 16. Android 端验收标准

满足以下条件即可认为一期 Android 端完成：

1. 用户可从搜索入口进入语音搜索页
2. 页面显示“按住并说出物品名称或品牌”
3. 按住时开始录音并实时显示 partial 文本
4. 松开后出现“正在识别并搜索”
5. 最终成功展示与文字搜索一致的结果列表
6. 权限拒绝、网络失败、识别失败时有明确提示

---

## 17. 最终建议

Android 客户端一期应坚持“**轻客户端、重服务端**”原则：

- 客户端负责交互、录音、实时回显、展示结果
- 服务端负责真正的 STT、最终文本、搜索逻辑

这样能最大化复用当前 Kotlin + Compose + ViewModel + Repository 架构，同时避免在一期把客户端做得过于复杂。

## 附：Envelope 解析建议

Repository 层应统一处理 Envelope，而不是把 UI 直接暴露给裸网络模型。

建议逻辑：

1. 调用 `VoiceSearchApi.finalizeVoiceSearch()`
2. 判断 `envelope.code == 0 && envelope.data != null`
3. 成功时取 `envelope.data`
4. 失败时抛出业务异常或转为 `UiState.Error(envelope.message)`

伪代码示例：

```kotlin
suspend fun finalizeVoiceSearch(request: VoiceFinalizeRequest): VoiceFinalizeResponse {
    val envelope = api.finalizeVoiceSearch(request)
    if (envelope.code != 0 || envelope.data == null) {
        throw VoiceSearchException(envelope.message)
    }
    return envelope.data
}
```
