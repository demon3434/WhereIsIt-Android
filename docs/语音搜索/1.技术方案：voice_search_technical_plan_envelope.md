# WhereIsIt 语音搜索一期技术方案

## 1. 目标与范围

## 1.1 接口返回规范补充（Envelope）

本方案中的 **HTTP 接口返回** 均需符合 WhereIsIt 现有的 **Envelope 规范**。也就是说，业务数据不能直接裸返回，而应放在统一的外层包裹结构中。本文档后续凡是出现“响应结构”“响应示例”，如无特别说明，均表示 **Envelope.data** 中的业务载荷示例。

建议统一采用如下抽象结构：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "...": "业务数据"
  }
}
```

约束建议：

- `code`：业务状态码，成功为 `0`，失败使用项目既有错误码体系
- `message`：对本次请求结果的简短说明
- `data`：真正的业务响应体；成功时返回对象或列表，失败时可为 `null`

> 说明：WebSocket 的 `partial / finalizing / session` 等消息通常不走 HTTP Envelope；Envelope 规范主要约束 HTTP API 返回体。如果项目内部也要求 WebSocket 文本帧统一包裹，可额外在 WebSocket 消息中增加 `code`、`message` 字段，但这不属于一期必须项。


本文档描述 WhereIsIt 语音搜索一期的落地方案。目标是在尽量少改动现有系统的前提下，为 Android 客户端增加“按住说话搜索物品”的能力。

一期范围只覆盖以下流程：

1. 用户在 Android 客户端点击“语音搜索”按钮。
2. 弹出语音搜索页，提示“按住并说出物品名称或品牌”。
3. 用户按住按钮说话。
4. 一阶段实时识别服务使用 **sherpa-onnx / k2-fsa** 做流式识别，把中间文本实时回显到页面。
5. 用户松开按钮，表示输入完成。
6. 服务端把本次完整音频交给 **FunASR / Fun-ASR-Nano 路线的非流式更强模型** 做最终识别。
7. 服务端对最终文本做轻量归一化、切词/关键词提取、查询。
8. Android 客户端沿用现有文字输入搜索结果页的展示方式展示物品列表。

> 一期不追求“完整语音理解系统”，也不强依赖大型业务字典。优先做通路打通、体验可用、后续可迭代。

## 2. 设计原则

### 2.1 先快后准

一期采用“两阶段 ASR”：

- **第一阶段：流式、低延迟、负责回显**
  - 工具：sherpa-onnx
  - 作用：边说边转文字，给用户即时反馈
  - 目标：用户说话时页面持续有文本更新

- **第二阶段：非流式、准确率更高、负责最终查询**
  - 工具：FunASR / Fun-ASR-Nano 非流式识别
  - 作用：在用户松开后，对整段音频做最终识别，作为真正的查询输入
  - 目标：最终查询用的文本比第一阶段更稳

这种设计符合 sherpa-onnx 当前对流式与非流式 ASR、Android、WebSocket server/client 的支持能力，也符合 FunASR 当前对 ASR、VAD、文本后处理以及非流式强模型路线的定位。sherpa-onnx 官方说明其支持 streaming 和 non-streaming STT、Android、WebSocket server/client；FunASR 官方说明其提供 ASR、VAD、文本后处理等能力，并持续维护实时转写与离线转写服务。 citeturn966714view3turn966714view2turn966714view4

### 2.2 一期不做过重的“实时纠错”

用户希望“遇到同音词之类，需要实时自行修正”，但一期建议不要在第一阶段做复杂的业务纠错，而采用以下更稳妥的做法：

1. 第一阶段只做：
   - 基础文本归一化
   - 少量高频规则修正
   - 热词增强
2. 第二阶段再产出最终文本。
3. 搜索侧再做一次轻量归一化与多路召回。

也就是说：

- **第一阶段负责“看起来像对的”**
- **第二阶段负责“真正拿去查库的”**

### 2.3 复用现有文字搜索能力

WhereIsIt 后端已经具备物品查询能力，Android 客户端也已有文字搜索结果展示能力。因此一期不单独设计新的结果页，只新增一条“语音转文本 -> 调现有搜索接口”的输入链路。

WhereIsIt 服务端 README 已说明系统当前支持物品、位置、分类、标签等管理能力，使用 FastAPI + SQLAlchemy + PostgreSQL；Android 客户端 README 说明客户端已具备“物品查询与展示”、“Repository 分层”、“Retrofit 网络请求封装”。 citeturn781242view0turn781242view3turn781242view4turn781242view5

## 3. 选型说明

## 3.1 第一阶段：sherpa-onnx

选择原因：

1. 支持 **streaming ASR**。
2. 支持 **Android**。
3. 支持 **WebSocket server/client**，适合本项目的“Android 录音 -> 服务端流式识别”模式。
4. 支持 **hotwords / contextual biasing**、VAD、endpoint detection，适合短语音搜索。
5. 支持本地和服务端部署，不依赖第三方在线接口。

sherpa-onnx 官方仓库与文档说明其支持 streaming / non-streaming STT、Android、WebSocket server/client、VAD 等能力。 citeturn966714view3turn267426search1turn267426search16

### 3.2 第二阶段：FunASR / Fun-ASR-Nano

选择原因：

1. 中文识别能力强。
2. 支持中文方言与地区口音覆盖。
3. 支持非流式高精度识别。
4. 与一期“松开后做最终识别”的交互完全匹配。

FunASR 官方仓库说明其提供 ASR、VAD、文本后处理等能力；Fun-ASR 路线强调对中文、方言、区域口音的支持，并持续提供离线转写与实时转写服务。 citeturn966714view2turn966714view4

### 3.3 为什么不是“全部只用一个模型”

如果只用第一阶段模型：

- 实时性好
- 但最终文本准确率一般

如果只用第二阶段模型：

- 最终文本更准
- 但用户按住说话时没有即时反馈，体验会差

因此一期用“两阶段 ASR”是更平衡的方案。

## 4. 一期总体架构

```text
Android 客户端
  ├─ 录音页（按住说话）
  ├─ 实时文本回显
  ├─ 松开后请求最终搜索
  └─ 复用现有结果页

           │
           ▼
WhereIsIt 服务端（FastAPI）
  ├─ 语音会话接口
  ├─ 流式 STT 网关（WebSocket）

  ├─ 非流式最终识别服务
  │    └─ FunASR / Fun-ASR-Nano
  ├─ 查询归一化 / 轻量分词
  └─ 复用现有物品搜索服务

           │
           ▼
PostgreSQL
  └─ 现有物品表 / 品牌字段 / 名称字段 / 标签字段
```

## 5. 语音搜索的完整时序

## 5.1 实时阶段

1. 用户进入语音搜索页。
2. 页面显示提示文案：**“按住并说出物品名称或品牌”**。
3. 用户按住录音按钮。
4. Android 开始采集 16kHz、单声道 PCM。
5. Android 建立 WebSocket，会话开始。
6. Android 按帧上传音频给服务端。
7. 服务端把音频喂给 sherpa-onnx streaming recognizer。
8. 服务端持续把 partial / intermediate text 回传给客户端。
9. 客户端实时显示文本。

## 5.2 松开后的最终阶段

1. 用户松开按钮。
2. Android 发送 `stop` 信号，同时上传剩余缓冲音频。
3. 服务端关闭 streaming 输入。
4. 服务端把本次音频缓冲作为整段音频，交给 FunASR / Fun-ASR-Nano 非流式识别。
5. 服务端拿到最终文本后执行：
   - 文本归一化
   - 轻量切词/关键词提取
   - 搜索
6. 服务端返回：
   - final_text
   - normalized_query
   - items[]
7. Android 跳转或刷新到现有搜索结果列表。

## 6. 一期接口设计建议

## 6.1 WebSocket：实时识别

建议新增：

- `WS /api/voice-search/stream`

职责：

- 建立一次语音搜索会话
- 接收音频块
- 回传实时文本
- 接收 stop 事件
- 在会话结束后返回 final job id 或直接返回最终结果摘要

### 客户端发送消息示例

```json
{ "type": "start", "sample_rate": 16000, "format": "pcm_s16le" }
```

```json
{ "type": "audio", "seq": 1, "data": "<base64 pcm bytes>" }
```

```json
{ "type": "stop" }
```

### 服务端回传消息示例

```json
{ "type": "partial", "text": "小米" }
```

```json
{ "type": "partial", "text": "小米 note" }
```

```json
{ "type": "partial", "text": "小米 note 手机" }
```

```json
{ "type": "finalizing", "message": "正在识别并搜索" }
```

## 6.2 HTTP：最终搜索

如果不想把所有逻辑都塞进同一个 WebSocket，也可以拆成：

- WebSocket 只负责第一阶段实时识别
- HTTP 只负责第二阶段最终识别与搜索

建议新增：

- `POST /api/voice-search/finalize`

请求：

- audio file 或 session_id
- optional first_stage_text

响应：

```json
{
  "final_text": "红米note8手机",
  "normalized_query": "红米 note8 手机",
  "keywords": ["红米", "note8", "手机"],
  "items": [
    {
      "id": 123,
      "name": "note手机",
      "brand": "小米"
    }
  ]
}
```

## 7. 文本处理策略

一期不依赖大型业务字典，因此文本处理采用“轻规则 + 复用已有数据”的方式。

## 7.1 第一步：归一化

对第一阶段与第二阶段文本都执行：

- 中文全角半角统一
- 英文大小写统一
- 去多余空格
- 数字归一化（如 `note 8` -> `note8`）
- 去语气词（如“帮我找一下”、“那个”、“请帮我找”）

## 7.2 第二步：轻量切词 / 关键词提取

一期建议不要把“通用分词”当核心，而采用以下顺序：

1. 优先识别数据库中已有品牌名
2. 再识别数据库中已有物品名中的高频片段
3. 再用轻量分词库做兜底

也就是：

- 品牌：优先从物品现有 brand 字段抽取
- 名称片段：优先从 item_name 字段抽取
- 分词器：只做辅助

## 7.3 第三步：多路查询

为提高召回，一期可以对同一条最终文本构造多种查询：

1. 原始最终文本
2. 归一化文本
3. 去语气词文本
4. 只保留品牌与名称关键词的文本

然后把多路结果合并、去重、排序。

## 8. 实时“同音词修正”的一期策略

用户要求“一阶段实时解析时，遇到同音词之类，需要实时自行修正”。

一期建议把这件事拆成三层：

### 8.1 第一层：热词增强

把已有品牌、常见物品名、常见分类名作为 sherpa-onnx 热词输入，优先改善第一阶段回显效果。

### 8.2 第二层：极少量高频实时规则

只做最常见、最稳定的修正，例如：

- `note 8` -> `note8`
- `red mi` -> `redmi`
- 去掉重复词尾

### 8.3 第三层：以第二阶段结果为准

不要要求第一阶段完成全部业务纠错。最终查询与最终展示以第二阶段结果为准。

> 一期中，“实时修正”更适合作为视觉体验增强，而不是作为最终业务真值。

---

## 9. 搜索策略

一期建议只在服务端新增一个“语音搜索专用查询入口”，内部仍然尽量复用现有文字搜索逻辑。

### 9.1 语音搜索查询逻辑

输入：

- `final_text`
- `normalized_query`
- `keywords`

处理：

1. 优先按品牌和物品名关键词查询
2. 如果结果过少，再做模糊匹配
3. 结果按以下顺序排序：
   - 品牌命中
   - 名称完整包含
   - 名称部分包含
   - 其他相关字段匹配

### 9.2 兼容“录入不标准”

对于“红米note8手机”这种查询，而数据库中可能是：

- 品牌：小米
- 名称：note手机

一期不强求完全解决所有品牌家族与型号归并问题，但建议先做以下弱版本：

1. 对常见品牌建立少量别名映射配置
2. 对常见型号数字做归一化
3. 允许名称字段模糊匹配

也就是说，一期先做“80% 常见可用”，后续再补品牌家族、alias 表、搜索派生文档。

## 10. 性能与资源建议

### 10.1 服务器部署建议

- sherpa-onnx streaming 服务：常驻进程
- FunASR 非流式模型：常驻进程，避免每次冷启动
- FastAPI：通过内部 service 调用这两个模型服务

### 10.2 音频建议

- 采样率：16kHz
- 声道：mono
- PCM：16-bit
- 单次语音搜索建议控制在 1~5 秒

### 10.3 延迟目标

一期可接受的体验目标：

- 按住说话期间：实时有 partial 文本
- 松开后 0.5~2 秒内返回最终列表

> “1 秒内”作为优化目标可以保留，但一期文档建议用“尽量压缩到 1 秒左右”而不是写成强 SLA。

## 11. 日志与可观测性

为了后续迭代，需要记录：

- session_id
- 第一阶段 partial 数量
- 第一阶段 final text
- 第二阶段 final text
- 最终 query
- 搜索命中结果数
- 用户最终点击了哪个 item
- 音频时长
- 第一阶段耗时 / 第二阶段耗时 / 搜索耗时

这将直接决定后续是否要继续做：

- 更强的字典
- 品牌 alias
- 实时纠错规则
- 搜索排序优化

以下内容不建议在一期实现：

1. 完整业务字典系统
2. 品牌家族与型号知识图谱
3. 实时多候选改写 UI
4. 端侧离线 STT
5. 复杂语义理解
6. 录音中断后断点续传
7. 多轮语音交互

## 14. 里程碑建议

### M1：打通链路
- Android 录音 + WebSocket
- sherpa-onnx partial 文本回显
- 松开后提交 finalize
- 服务端返回最终文本

### M2：接入搜索
- 服务端 finalize 后直接调现有搜索
- Android 复用现有结果页

### M3：体验优化
- 热词增强
- 文本归一化
- 常见同音词修正规则
- 埋点与日志

---

## 15. 最终建议

WhereIsIt 语音搜索一期应采用“**流式回显 + 非流式最终识别 + 复用现有搜索结果页**”的最小可用方案：

- 第一阶段：sherpa-onnx，解决“用户说话时能看到文字”
- 第二阶段：FunASR / Fun-ASR-Nano，解决“最终拿去查库的文本更准”
- 搜索阶段：轻量归一化 + 复用现有查询接口

这样可以在不引入大而全字典系统的前提下，先把最重要的语音搜索链路落地，并为后续品牌 alias、型号归一化、纠错词典等迭代留出空间。
