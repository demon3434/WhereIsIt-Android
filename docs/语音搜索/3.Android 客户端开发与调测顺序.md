# WhereIsIt 语音搜索：Android 客户端开发与调测顺序

本文面向 **WhereIsIt Android 客户端开发人员**，目标是在服务端尚未全部完成时，也能先把语音搜索入口、录音交互、实时字幕和最终结果展示跑通。

---

## 1. 开发目标

一期客户端需要完成：

1. 在合适位置提供“语音搜索”入口
2. 弹出语音输入页，提示：**按住并说出物品名称或品牌**
3. 按住期间采集音频，并通过 WebSocket 发送给服务端流式接口
4. 实时接收字幕并显示在界面上
5. 用户松开后，上传完整音频到非流式最终接口
6. 用最终接口返回的物品列表，复用已有文字搜索结果展示方式

---

## 2. 推荐开发顺序

### 阶段 A：先做 UI 和页面状态机

先不要等服务端完成。先把页面、按钮、状态流转写好。

建议页面状态：

- `Idle`：未开始
- `Recording`：按住录音中
- `Recognizing`：已松开，等待最终结果
- `Success`：展示结果
- `Error`：展示错误

建议先完成：

- 搜索页语音按钮入口
- 弹出层/页面布局
- 提示文案
- 按住、松开、取消等交互
- 实时字幕区域
- 错误提示区域

#### 一期文案建议

- 初始：`按住并说出物品名称或品牌`
- 录音中：`正在聆听...`
- 松开后：`正在识别并搜索...`
- 失败：`语音识别失败，请重试`

---

### 阶段 B：先接 mock 数据，把页面跑通

在服务端真实接口完成前，先做本地 mock 或对接服务端 mock。

#### B1. mock WebSocket 结果
模拟实时字幕：

- 300ms 后显示：`红米`
- 600ms 后显示：`红米note`
- 900ms 后显示：`红米note8手机`

#### B2. mock 最终 HTTP 结果
模拟返回 Envelope：

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "recognizedText": "红米note8手机",
    "normalizedText": "红米note8手机",
    "items": []
  }
}
```

先把：
- 录音后进入 loading
- 收到结果后跳转/展示列表

这条链路跑通。

---

### 阶段 C：实现录音与音频缓存

#### C1. 先做本地录音，不接网络
先独立完成：

- 麦克风权限申请
- AudioRecord 初始化
- 16kHz / mono / PCM 采集
- 按住开始录音
- 松开停止录音
- 录音数据缓存到内存或临时文件

注意：

- WebSocket 流式发送需要音频帧缓存
- 最终非流式接口还需要完整音频

所以一期推荐：

- 一边录音一边把小帧发给流式接口
- 同时把完整数据缓存在本地
- 松开后把完整音频上传到最终接口

#### C2. 优先保证音频格式一致
与服务端统一：

- 16kHz
- 单声道
- PCM 16-bit

---

### 阶段 D：接入最终 HTTP 接口

建议优先接这个接口，因为它决定最终功能是否可用。

#### D1. 定义 Envelope 模型
例如：

```kotlin
data class Envelope<T>(
    val code: Int,
    val message: String,
    val data: T?
)
```

#### D2. 定义最终搜索返回模型
例如：

```kotlin
data class VoiceSearchFinalData(
    val recognizedText: String,
    val normalizedText: String,
    val items: List<ItemDto>
)
```

#### D3. Repository 处理方式
流程建议：

1. 上传完整音频
2. 解析 `Envelope<VoiceSearchFinalData>`
3. 判断 `code == 0`
4. 取 `data.items`
5. 复用现有文字搜索结果展示逻辑

#### D4. 先接 mock，再接真实接口
推荐顺序：

1. Retrofit 对接 mock 最终接口
2. 真机跑通结果页
3. 再切换真实服务端

---

### 阶段 E：接入流式 WebSocket 接口

在最终 HTTP 跑通后，再接 WebSocket。

#### E1. 连接生命周期
按住时：

1. 建立 WebSocket
2. 发送 `start`
3. 持续发送音频帧
4. 接收 partial 文本并刷新 UI

松开时：

1. 发送 `stop`
2. 关闭流式发送
3. 触发最终 HTTP 上传

#### E2. UI 更新原则
实时字幕只是提示，不要把它当最终搜索词。

真正执行搜索时，应以最终 HTTP 接口返回的：

- `recognizedText`
- 或 `normalizedText`

为准。

#### E3. 先接 mock websocket
不要一开始就连真实 sherpa-onnx 服务。先用 mock 验证：

- partial 是否实时刷新
- stop 时机是否正确
- 页面状态是否稳定

---

### 阶段 F：联调与细节优化

重点处理：

- 用户松开后是否立即停止录音
- 网络慢时的 loading 状态
- 识别失败后的重试
- 页面返回时资源释放
- WebSocket 断开重连
- 横竖屏/页面切换时状态恢复

---

## 3. 推荐调测顺序

### 3.1 本地 UI 调测
先不接任何真实网络：

- 语音页显示
- 按住按钮动画
- 字幕区域刷新
- loading / error / success 状态切换

### 3.2 本地录音调测
验证：

- 权限正常
- 音频能采到
- 停止时能拿到完整 buffer
- PCM 参数和服务端约定一致

### 3.3 最终 HTTP 联调
优先顺序：

1. mock 最终接口
2. 真实最终接口

这是最重要的联调链路。

### 3.4 WebSocket 联调
优先顺序：

1. mock websocket
2. 真实 websocket

---

## 4. 客户端开发注意事项

### 不要先做的事情
一期先不要做：

- 客户端本地分词
- 客户端本地品牌词典
- 客户端本地同音词纠错系统
- 独立于文字搜索的新结果页

### 一期应坚持的原则

- 实时字幕只负责展示
- 最终搜索结果以服务端返回为准
- 优先复用现有文字搜索列表 UI
- 优先把页面链路做稳定

---

## 5. 交付检查清单

客户端一期完成时，至少应满足：

- [ ] 有语音搜索入口
- [ ] 有语音输入页面/弹层
- [ ] 有按住说话交互
- [ ] 能采集 16k 单声道 PCM
- [ ] 能缓存完整音频
- [ ] 能对接最终 HTTP 接口
- [ ] 能解析 Envelope
- [ ] 能展示服务端返回的物品列表
- [ ] 能对接流式 WebSocket 并显示 partial 文本
- [ ] 页面返回和异常时能释放资源

## 6. 与服务端的协作结论

Android **不需要等服务端全部完成后再开发**。

推荐协作方式：

1. **先和服务端一起冻结协议**
   - HTTP 请求字段
   - Envelope 响应结构
   - WebSocket 消息格式

2. **基于 mock 并行开发**
   - 服务端提供 mock 接口
   - Android 先接 mock 做 UI、状态机、录音、解析

3. **分两次联调**
   - 第一次：最终 HTTP 接口联调
   - 第二次：流式 WebSocket 联调

也就是说：

- 可以并行开发
- 不建议“服务端全部做完，再开始 Android”
- 但必须先锁定接口契约

---

## 7. 推荐的并行开发策略

### 第 1 周

服务端：
- 冻结接口文档
- 提供 mock 最终接口
- 提供 mock websocket

Android：
- 完成语音页 UI
- 完成状态机
- 接 mock 跑通页面

### 第 2 周

服务端：
- 接 FunASR-Nano
- 打通真实最终接口

Android：
- 接真实最终接口
- 跑通结果展示

### 第 3 周

服务端：
- 接 sherpa-onnx 流式

Android：
- 接真实 websocket
- 优化实时字幕体验

这个顺序返工最少，也最适合一期目标。
